1 · Deepen Static Resolution into one guarded pipeline

Strong
src/parser.js (482) bin/generate.mjs:173

Problem

Static Resolution — the layer ADR-0002 demands be exact — is a fixed but implicit pipeline: include-merge → !reference → extends → defaults, hard-coded inside parsePipeline (lines 361/367/400/407) with nothing enforcing the order. One merge primitive, mergeGitlab (272–281), is overloaded for three unrelated jobs: extends-merge, include-merge, and multi-doc spec: merge. The roadmap's first task — fix include-merge to GitLab's "main file overrides included top-level keys" rule — must change include-merge without touching extends-merge semantics. With one shared primitive, that fix has no locality: it lands in the blast radius of everything. Error reporting leaks past the interface too: warnings/unresolved accumulate by reference-threading side effects, and on failure the unresolved list rides an out-of-band Error.unresolved property that generate.mjs:173 must simply know about.

Solution

Keep the small external interface (one resolve call), but restructure the implementation around internal seams per phase: separate merge primitives for include-merge vs extends-merge vs spec-merge, an explicit ordered phase list, and results (model + warnings + unresolved) returned as values instead of side-effect accumulation and thrown-error side channels. The oracle harness becomes the CI gate at the external seam.

Benefits

Locality: the ADR-0002 merge fix becomes an edit to one primitive with its own tests — extends/spec provably untouched. Leverage: callers (generator, viewer, oracle, tests) keep a one-function interface while gaining a value-typed error contract. Tests: each phase gets unit tests at its internal seam; the existing 342-line parser suite keeps guarding the external one.

2 · Gather Simulation behind one seam (before Scenario Profiles land)

Strong
src/refs.js (417) src/main.js:59–83 src/render.js:136,156 src/drawer.js:3

Problem

The verdict — Simulation's core output — is smeared across three files via two mechanisms: job verdicts computed in refs.js, rendered as node badges in render.js; the sibling workflow verdict from the same filterModel call rendered as prose in main.js:76–83. Meanwhile formatCondition — pure presentation — is parked in the evaluator because it wants tokenize(). Roadmap task #2 adds user-defined Scenario Profiles: today that feature would touch four files.

Solution

Deepen refs.js into the Simulation module: one call takes a model + a Scenario Profile and returns everything the UI needs — per-job verdicts, the workflow verdict, and display-ready condition strings — so no caller re-derives or re-splits verdict data. The six built-in REF_CONTEXTS become built-in Scenario Profiles: adapters at the same seam the committed YAML profiles will use. That's two adapters — a real seam, per the one-adapter/two-adapter rule, and the profile feature becomes "add a loader adapter," not "thread a new concept through four files."

Benefits

Locality: ADR-0001's "always labeled as assumptions" invariant gets one enforcement point instead of three render paths. Leverage: Scenario Profiles, provenance labels, and future what-if UI all arrive through one interface. Tests: the excellent refs.test.mjs suite extends naturally; untested formatCondition/contextCounts get covered as part of the verdict payload.

3 · Turn serialize.js into a real Manifest module

Worth exploring
src/serialize.js (24) manifest.schema.json bin/generate.mjs:240 src/main.js:351,358

Problem

serialize.js is textbook shallow: two one-object-literal functions doing Map⇆array. It barely passes the deletion test today — its only value is being the single home of the manifest shape. But the grilling promoted the Manifest to a versioned public contract (prerequisite for Manifest Diff), and nothing owns that contract: no schema_version, no validation against manifest.schema.json, deserializeModel untested, and every job's raw resolved YAML fans out into the artifact unreviewed (parser internal representation leaking to disk).

Solution

Deepen it into the Manifest module: owns schema_version stamping (plus oracle-verified GitLab version per ADR-0002), validates on read and write, and makes the raw-YAML-in-manifest choice an explicit, documented policy. Manifest Diff later lands inside this module — comparison logic next to the contract it compares.

Benefits

Locality: schema evolution and version-mismatch handling in one file instead of scattered across generator and viewer. Leverage: generator, viewer, and future diff CLI all consume one contract. Tests: round-trip (serialize→deserialize) and schema-validation tests become natural; today deserializeModel has zero coverage.

4 · Give the needs-edge one home (and graph/layout their first tests)

Worth exploring
src/parser.js:308–330,417 src/graph.js:33–49 src/layout.js src/render.js:155–162

Problem

Understanding one DAG edge takes four files: parser defines tri-state needs (null = stage-ordered, [] = immediate, [...] = explicit), graph.js re-encodes that convention into implicit stage edges, layout positions them, render badges them. Every consumer must re-learn the tri-state invariant — the parser's interface is bigger than its signature. And the two pure modules in the chain, graph.js and layout.js, have zero tests: no test file imports them; they're only reachable through a full parsed model.

Solution

Make edge derivation the graph module's whole job: it alone consumes the tri-state and emits edges tagged with why they exist (explicit need vs stage order), so layout and render stop caring which kind they got. Add fixture-based tests through buildGraph/layoutPipeline's own interfaces — both are pure functions begging for them.

Benefits

Locality: the edge rule lives in one file; a GitLab needs semantics change is a one-module edit. Tests: the currently-invisible implicit-edge rule ("nearest earlier non-empty stage") gets pinned by unit tests before Manifest Diff starts comparing edges — diff correctness depends on edge derivation being right.
